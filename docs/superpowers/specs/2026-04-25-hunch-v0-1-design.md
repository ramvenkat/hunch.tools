# Hunch v0.1 Design

## Purpose

Hunch is a local CLI for PMs who code. It turns a customer problem, persona, and target user action into a disposable working prototype that can be shown to customers quickly.

The v0.1 goal is the full loop:

```text
hunch new -> hunch run -> hunch ask -> hunch decide -> hunch show
```

The product is intentionally not a production app builder. It is a research instrument for testing one hunch with one spike, then learning from the customer response.

## Scope

v0.1 includes:

- A standalone TypeScript npm CLI package with a `hunch` executable.
- Local global state under `~/.hunch`.
- Per-spike state under `~/hunches/<date-slug>/.hunch`.
- A packaged Vite + React + TypeScript prototype template.
- An Anthropic-backed agent runtime with scoped tools.
- Commands for `new`, `run`, `ask`, `decide`, `show`, `list`, and `open`.

v0.1 excludes:

- Multiple model providers.
- Hosted sharing or deployment.
- Windows support beyond best effort.
- Telemetry.
- Plugin systems or custom tool registration.
- Production hardening of generated prototypes.

## Architecture

The repo will be organized as a TypeScript CLI with four major layers:

```text
src/
  cli.ts
  commands/
  state/
  agent/
  tools/
  prompts/
  template/
templates/
  app/
```

The command layer owns terminal I/O and delegates to services. The state layer owns global config, active spike tracking, and spike file helpers. The agent layer owns Anthropic request composition, streaming, tool dispatch, and session persistence. The template layer owns the packaged Vite app copied into new spikes.

This keeps the CLI commands thin and lets the agent/tool code be tested without running interactive prompts.

## Commands

### `hunch new`

`new` asks for the customer problem, persona, and one target action. It creates a spike directory named `YYYY-MM-DD-<slug>`, writes `.hunch/problem.md`, `.hunch/persona.md`, `.hunch/journey.md`, `.hunch/decisions.md`, `.hunch/session.jsonl`, and `.hunch/config.yaml`, copies the Vite app template into `app/`, installs dependencies, and runs the initial agent generation.

The spike slug is generated by a focused one-shot model call, with a fallback local slugifier if the API call fails.

### `hunch run`

`run` reads `~/.hunch/active`, starts `npm run dev` in the active spike's `app/` directory, streams logs, opens the browser, and shuts down cleanly on Ctrl-C.

### `hunch ask`

`ask` loads the active spike context, recent session history, decisions, and a shallow app file tree. It calls the Anthropic agent loop with scoped tools and persists the full turn to `session.jsonl`.

If no message is provided, it enters an interactive prompt loop.

### `hunch decide`

`decide` reads `.hunch/decisions.md` and offers an interactive review flow. The user can approve decisions, push back on a decision, or remove a decision. Pushback re-enters the agent with the selected decision as context.

### `hunch show`

`show` prepares the spike for a customer interview. It generates realistic seed data, writes `.hunch/show/script.md` and `.hunch/show/questions.md`, starts the dev server in demo mode, prints the walkthrough and questions, and waits for user input before stopping the server.

### `hunch list` and `hunch open`

`list` prints spikes from the configured spike directory and marks the active one. `open <name>` updates `~/.hunch/active`.

## Agent Runtime

The main system prompt lives in `src/prompts/main.md`. It frames Hunch as a prototyping agent for PMs who code, prioritizing the customer problem, persona, and target journey over production concerns.

The runtime loop:

1. Load spike context.
2. Compose the system prompt.
3. Send recent session messages and the current user request to Anthropic.
4. Stream text and tool calls.
5. Execute scoped tools.
6. Continue until the model ends the turn.
7. Persist assistant output, tool calls, and tool results to `session.jsonl`.

The first implementation will be Anthropic-only and will use the official SDK directly. Provider abstraction is out of scope for v0.1.

## Tools

The agent tools are deliberately narrow:

- `read_file`: read a file within the active spike.
- `write_file`: write a file within the active spike.
- `edit_file`: perform validated string replacement within the active spike.
- `list_files`: list files within the active spike, depth-limited.
- `run_shell`: run allow-listed commands only.
- `decide`: append a UX decision with rationale to `.hunch/decisions.md`.
- `generate_seed_data`: create realistic demo content for the spike.
- `push_back`: classify scope creep against the current journey.

Every file path is resolved and checked to stay inside the active spike directory. The agent cannot read or write generated dependency/build directories or Vite config files because those are executable surfaces for later CLI commands. `run_shell` only permits commands needed for v0.1 prototype workflows. Package-script execution is treated as untrusted because the agent can edit app files, so the shell tool permits `npm install` only with lifecycle scripts disabled.

## Data Flow

Global state:

```text
~/.hunch/
  config.yaml
  active
```

Spike state:

```text
~/hunches/YYYY-MM-DD-slug/
  .hunch/
    problem.md
    persona.md
    journey.md
    decisions.md
    session.jsonl
    config.yaml
    show/
      script.md
      questions.md
  app/
  README.md
```

The `.hunch` files are agent-managed and concise. The `app/` directory is normal user-owned Vite code. Hunch may edit it through tools, but the user can edit it directly at any time.

## Error Handling

The CLI should fail clearly and recoverably:

- Missing active spike: explain how to run `hunch list` or `hunch open`.
- Missing API key: name the expected env var and config key.
- Anthropic API failure: preserve local state and print the failed operation.
- Tool path violation: reject the operation and report that paths must stay inside the spike.
- Non-unique edit string: reject the edit and ask the agent to read the file first.
- Dev server startup failure: stream the underlying npm error.

Prototype app error handling remains intentionally light because generated apps are disposable.

## Testing

Verification will combine automated checks and a manual end-to-end spike:

- TypeScript typecheck.
- Unit tests for path scoping, config loading, active spike resolution, session JSONL, and edit validation.
- CLI smoke tests for `list`, `open`, and non-agent parts of `new`.
- Manual local test of `new -> run -> ask -> decide -> show` with an Anthropic API key.

Agent API calls should stay behind a small wrapper so most tests do not require network access or paid model calls.

## Implementation Strategy

Although the target is the full v0.1, implementation will be milestone-gated:

1. Repo and package scaffold.
2. CLI command shell and state helpers.
3. Vite app template and spike creation.
4. Dev server runner.
5. Agent runtime and scoped tools.
6. `ask` end-to-end.
7. Decisions and pushback.
8. `show` demo preparation.
9. Build, smoke test, and README polish.

Each milestone should leave the repo runnable before moving to the next one.

## Implementation Defaults

- The npm package name is `hunch-cli`; the installed binary is `hunch`.
- `hunch ask` prints concise progress by default and exposes verbose tool-call output behind `--verbose`.
- `hunch decide` preserves history by marking decisions as approved, superseded, or removed instead of deleting decision text.
- The initial app template includes Tailwind and shadcn-compatible local UI components. It does not require `npx shadcn` during `hunch new`.

These defaults can change later, but v0.1 implementation should treat them as decided.
